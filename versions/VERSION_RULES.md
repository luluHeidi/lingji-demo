# Demo 版本规则 VX.Y.Z

> 本规则仅服务于产品 demo 的版本管理，无需与线上系统版本号保持一致。

---

## 版本号定义

### X — 大版本（S1 验收封版）

- S1 工作流全部 M 文件开发 + 验收通过时，X + 1
- 封版动作：将当前工作目录完整快照到 `versions/VX.0.0/`
- 封版同时读取线上代码仓库最新 tag，记入**版本对应表**
- 封版后 `VX.0.0/` 目录**只读不改**
- 一般发生在正式系统代码上线后

### Y — M 文件序号（开发推进）

- 每个 M 文件对应一个固定的 Y 值（Y = M 文件编号）
- M1 → Y=1，M2 → Y=2，...，M9 → Y=9
- 进入新 M 文件开发时，Z 重置为 0
- 新建 `versions/VX.Y.0/` 快照作为该 M 的初版

### Z — 过程修订（同一 M 内迭代调整）

- 同一 M 文件开发过程中，每次提交调整 Z + 1
- 查 Z 的最大值即该 M 的最新版本
- 每次修订都需要快照到 `versions/VX.Y.Z/`

### VX.0.Z — 上线后维护（S3）

- 上线封版后的维护修复走 Y=0 通道
- 例：V1.0.0 封版 → V1.0.1 修一个 bug → V1.0.2 又修一个

## 版本流转示例

```
一期开发中（X=0）：
V0.1.0  M1初版
V0.1.1  M1调整
V0.2.0  M2初版
...
V0.9.0  M9初版
V0.9.1  M9调整

一期封版：
V1.0.0  ← 一期全部验收通过，快照封版（只读）
         对应线上：BE v0.0.46 / FE v0.0.13

一期维护（S3）：
V1.0.1  修复 bug A
V1.0.2  修复 bug B

二期开发（并行）：
V1.1.0  二期M1初版
V1.1.1  二期M1调整
V1.2.0  二期M2初版
...

二期封版：
V2.0.0  ← 二期全部验收通过，快照封版
```

## 快照操作规范

### 何时快照

1. **M 文件初版完成**时：快照 `VX.Y.0`
2. **M 文件迭代调整后**（老板反馈修改完成时）：快照 `VX.Y.Z+1`
3. **大版本封版**时：快照 `VX+1.0.0`

### 快照内容

快照目录 `versions/VX.Y.Z/` 包含：
- `index.html` — 主页面
- `css/` — 框架样式
- `js/` — 框架脚本
- `modules/` — 所有功能模块（含 HTML + JS + CSS + JSON）
- `VERSION_NOTES.md` — 该版本的变更说明（可选）

### 快照操作

```bash
# 快照当前工作目录到指定版本
DEMO_BASE="demo/IP权益变现应用"
VERSION="V1.9.1"
mkdir -p "$DEMO_BASE/versions/$VERSION"
cp index.html "$DEMO_BASE/versions/$VERSION/"
cp -R css js modules "$DEMO_BASE/versions/$VERSION/"
```
