# 视频作品授权 - 授权管理 - 授权审查 前端需求文档

> **版本**: v1.0  
> **范围**: 授权审查首页（列表页）+ 审查详情页（二级页）  
> **适用对象**: 零业务背景经验的前端开发人员  

---

## 一、系统框架概述（目录级）

### 1.1 顶栏（Top Bar）

顶栏固定于页面顶部，高度 56px，背景深蓝色（`#1a1a2e`），包含以下 5 个一级导航 Tab：

| 序号 | Tab 名称 | 说明 |
|------|---------|------|
| 1 | IP资产全景图 | 其他模块，本文档不涉及 |
| 2 | IP权益引入和生产 | 其他模块，本文档不涉及 |
| 3 | **IP权益变现应用** | **当前激活 Tab**，点击后左侧展示侧边栏 |
| 4 | IP知识产权保护 | 其他模块，本文档不涉及 |
| 5 | IP数据服务 | 其他模块，本文档不涉及 |

**交互规则**：
- 仅「IP权益变现应用」Tab 激活时，左侧显示侧边栏菜单
- 其他 Tab 激活时，侧边栏隐藏，内容区全屏显示占位页

### 1.2 侧边栏（Sidebar）

宽度 220px，固定于左侧，灰色背景（`#f8f9fa`），包含 7 个菜单分组：

| 序号 | 菜单分组 | 子菜单项 |
|------|---------|---------|
| 1 | 视频作品授权 | **授权管理**（当前模块） |
| 2 | 游戏IP商业化 | 游戏授权管理 |
| 3 | 音乐IP商业化 | 音乐版权管理 |
| 4 | 衍生品管理 | 衍生品授权 |
| 5 | 品牌合作 | 品牌合作管理 |
| 6 | 内容营销 | 营销活动管理 |
| 7 | 数据分析 | 分析看板 |

**交互规则**：
- 菜单分组标题可折叠/展开
- 点击子菜单项时，动态加载对应模块到右侧内容区
- 当前激活的子菜单项高亮（蓝色左边框 + 蓝色文字 + 淡蓝背景）

### 1.3 内容区

位于顶栏下方、侧边栏右侧的主要区域，宽度 `calc(100% - 220px)`，背景色 `#f8f9fa`。模块通过动态加载（fetch HTML + 加载 CSS/JS）渲染到内容区。

## 二、授权审查首页（列表页）

### 2.1 页面结构

页面整体从上到下依次为：

```
[面包屑导航]

  ├─ 我的项目Tab内容（非本文档范围）
  └─ 授权审查Tab内容 ← 当前页面
       ├─ 分类区块：二创权益
       │   ├─ 区块头部（标题 + 添加审查IP按钮）
       │   ├─ 筛选栏
       │   ├─ 数据表格
       │   └─ 分页控件
       └─ 分类区块：分销权益
           ├─ 区块头部（标题）
           ├─ 筛选栏
           ├─ 数据表格
           └─ 分页控件
```

### 2.2 面包屑导航

- 位置：页面顶部，内容区内
- 内容：`授权管理`（固定文本）
- 样式：14px，灰色文字，底部 1px 分隔线

### 2.3 分类 Tab 栏

包含两个 Tab：
- **我的项目**：切换到项目列表视图（不在本文档范围）
- **授权审查**：切换到授权审查视图（当前页面）

**交互规则**：
- 点击 Tab 切换对应内容区域的显隐
- 激活 Tab 有蓝色底部指示线 + 蓝色文字
- 未激活 Tab 为灰色文字

### 2.4 授权审查内容区

#### 2.4.1 分类区块

数据按权益类型分为两个区块：
1. **二创权益**（type: `ugc`）—— 包含主要审查数据
2. **分销权益**（type: `distribution`）—— 包含分销相关审查数据

每个区块独立渲染，结构相同。

#### 2.4.2 区块头部

```
┌─────────────────────────────────────────────────────┐
│ [二创权益]                              [+添加审查IP] │
└─────────────────────────────────────────────────────┘
```

- **标题**：左对齐，16px，加粗，深色文字
- **添加审查IP按钮**：仅在「二创权益」区块显示，右对齐
  - 样式：蓝色主色按钮，带 `+` 图标
  - 点击打开「添加审查IP」弹窗（见 2.7 节）

#### 2.4.3 筛选栏

筛选栏为水平一行布局，包含 4 个下拉筛选器 + 1 个搜索框：

```
┌──────────────────────────────────────────────────────────────────────┐
│ [审核进度 ▼] [任务类型 ▼] [品类 ▼] [法务复审结论 ▼]    [🔍搜索作品名称/任务ID...] │
└──────────────────────────────────────────────────────────────────────┘
```

**筛选器规格**：

| 筛选器 | 默认值 | 选项 | 对应字段 |
|--------|--------|------|---------|
| 审核进度 | 全部进度 | 待初审、待复审、待终审、已确权 | reviewStatus（经映射） |
| 任务类型 | 全部类型 | 风险审查、新热确权、增补确权 | 由 externalAuthRights + listType 派生 |
| 品类 | 全部品类 | 动态收集（电视剧、动漫、综艺等） | category |
| 法务复审结论 | 全部结论 | 可用、不可用、权利瑕疵 | externalAuthRights |

**搜索框规格**：
- 位置：筛选栏最右侧
- placeholder：`搜索作品名称/任务ID...`
- 搜索方式：输入触发 300ms 防抖实时搜索
- 匹配规则：模糊匹配 workName、alias、taskId 三个字段

**筛选交互逻辑**：
1. 任一筛选器值变化时，立即触发数据过滤
2. 分页重置为第 1 页
3. 所有筛选条件取**交集**（AND 逻辑）
4. 筛选结果实时更新表格内容和分页

#### 2.4.4 审核进度映射规则

前端需将后端返回的 `reviewStatus` 字段映射为用户可见的进度文本：

| reviewStatus 原始值 | 显示文本 | 样式类 |
|---------------------|---------|--------|
| `权益初审待审` | 待初审 | `rr-progress-initial`（蓝色） |
| `权益复审待审` | 待复审 | `rr-progress-review`（橙色） |
| `运营授权评估待审` | 待终审 | `rr-progress-final`（紫色） |
| `已确权` | 已确权 | `rr-progress-confirmed`（绿色） |

#### 2.4.5 任务类型派生规则

任务类型不是直接存储的字段，而是由以下规则在前端实时计算：

```
if (item.externalAuthRights === '权利瑕疵' || item.externalAuthRights === '不可用') {
    任务类型 = '风险审查'
} else if (item.listType === '新热') {
    任务类型 = '新热确权'
} else {
    任务类型 = '增补确权'
}
```

**优先级**：法务复审结论的判定优先于 listType 判定。

#### 2.4.6 数据表格

表格列定义：

| 列名 | 字段 | 宽度 | 样式说明 |
|------|------|------|---------|
| 审核任务ID | taskId | 自适应 | 等宽字体(monospace)，可 title 悬停查看全文 |
| 作品名称 | workName | 自适应 | 超出省略号截断，title 悬停查看 |
| 别名 | alias | 自适应 | 空值显示 `-` |
| 品类 | category | 自适应 | 直接展示 |
| 版权ID | bqid | 自适应 | 等宽字体 |
| 任务发起时间 | taskInitTime | 自适应 | 格式：YYYY-MM-DD HH:mm:ss |
| 任务类型 | 计算值 | 自适应 | Tag 样式，三种颜色（见下表） |
| 法务复审结论 | externalAuthRights | 自适应 | 带颜色样式 |
| 审核进度 | reviewStatus(映射) | 自适应 | Tag 样式，四种颜色 |
| 操作 | — | 固定宽度 | 操作按钮 |

**任务类型 Tag 样式**：

| 任务类型 | CSS 类名 | 颜色 | 附加元素 |
|---------|---------|------|---------|
| 风险审查 | `rr-task-type-risk` | 红色系 | 带 ⚠ 警告图标 |
| 新热确权 | `rr-task-type-hot` | 橙色系 | 无 |
| 增补确权 | `rr-task-type-supplement` | 灰蓝色系 | 无 |

**法务复审结论样式**：

| 结论值 | CSS 类名 | 颜色 |
|--------|---------|------|
| 可用 | `ext-auth-available` | 绿色 |
| 不可用 | `ext-auth-unavailable` | 红色 |
| 权利瑕疵 | `ext-auth-defect` | 橙色 |

**操作列按钮**：
- 未确权项：显示「去审核」蓝色文字按钮
- 已确权项：显示「查看详情」灰色文字按钮

**行排序规则**：
- 「风险审查」类型的行优先展示（排在最前面）
- 「风险审查」行有特殊左侧红色边框标识（`rr-row-risk`）
- 已确权行有灰色淡化效果（`rr-row-confirmed`）

**点击行为**：
- 点击「去审核」或「查看详情」按钮 → 跳转到审查详情页（传入 item.id）

#### 2.4.7 分页控件

- 每页显示 **10 条**（`RR_PAGE_SIZE = 10`）
- 分页位于表格下方
- 包含：上一页按钮、页码按钮组、下一页按钮、总条数文本
- 页码按钮当前页高亮
- 首/尾页禁用对应按钮
- 点击页码触发数据重新渲染（不重新筛选，使用已筛选结果分页）

### 2.5 筛选条件联动流程

```
用户操作（下拉/搜索） 
    → 更新 rrFilters 对象中对应字段
    → 重置 rrUgcCurrentPage = 1
    → 调用 refreshRRWithFilters()
        → 遍历每个 category
            → filterRRItems(cat.items) 过滤
            → sortByTaskType(filtered) 排序（风险优先）
            → renderRRCards(type, sorted, page) 渲染
```

### 2.6 rrFilters 筛选状态对象

```javascript
const rrFilters = {
    progress: '',      // 审核进度下拉值（空 = 全部）
    taskType: '',      // 任务类型下拉值（空 = 全部）
    category: '',      // 品类下拉值（空 = 全部）
    extAuth: '',       // 法务复审结论下拉值（空 = 全部）
    searchText: ''     // 搜索关键词（空 = 不过滤）
};
```

### 2.7 「添加审查IP」弹窗

#### 2.7.1 弹窗结构

```
┌──────────────────────────────────────────────┐
│  添加审查IP                            [×]    │
│─────────────────────────────────────────────│
│  ⚠ 人工添加审查IP将直接进入审核队列...          │
│                                              │
│  版权ID（BQID）*  [          ]               │
│  作品名称         [自动填充区域]              │
│  待审查合同号     [          ]               │
│  初审瑕疵原因*    [                     ]    │
│                            0/500             │
│─────────────────────────────────────────────│
│                       [取消]  [确认添加]      │
└──────────────────────────────────────────────┘
```

#### 2.7.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 版权ID（BQID） | 文本输入 | ✅ | 输入后自动从作品库匹配 |
| 作品名称 | 自动填充 | — | 根据 BQID 自动填充作品名 + 品类，不可手动编辑 |
| 待审查合同号 | 文本输入 | 否 | 多条用 `；` 分隔 |
| 初审瑕疵原因 | 多行文本 | ✅ | 最大 500 字，带字数统计 |

#### 2.7.3 交互逻辑

1. **版权ID输入**：
   - 输入时实时从 `workLibrary` 数据中匹配 `bqid` 字段
   - 匹配成功 → 作品名称区域显示：`作品名 + 品类标签`（绿色填充态）
   - 匹配失败 → 作品名称区域显示：`未找到匹配作品，请确认版权ID是否正确`（红色未找到态）
   - 失焦时也执行一次匹配

2. **初审瑕疵原因**：
   - 打开弹窗时预填充系统中已有数据的瑕疵原因（从当前二创权益中找到最近一条有 systemDenyReason 的记录）
   - 输入时实时更新字数统计

3. **校验规则**：
   - 版权ID：必填，且必须在作品库中存在
   - 初审瑕疵原因：必填
   - 合同号关联性校验：如填写了合同号，其前缀必须与版权ID匹配
   - 校验失败：对应字段标红 + 字段下方显示错误提示

4. **确认添加**：
   - 校验通过后生成新审查条目
   - 自动生成 taskId（格式：`SQ` + 6位数字 + 2位小写字母）
   - 自动生成 taskInitTime（当前时间）
   - externalAuthRights 固定为 `权利瑕疵`
   - reviewStatus 固定为 `权益初审待审`
   - 新条目插入到二创权益列表**最前面**（unshift）
   - 关闭弹窗，刷新列表
   - 显示成功 Toast：`「{作品名}」已成功添加至授权审查队列`

5. **关闭弹窗**：
   - 点击右上角 × 按钮
   - 点击取消按钮
   - 点击遮罩层
   - 均关闭弹窗并重置表单

### 3.4 三级审核节点区域

#### 3.4.1 整体布局

```
┌──────────────────────────────────────────────────────┐
│  [左侧: 审核流程时间轴]  │  [右侧: 审核操作面板]       │
│  宽度: 240px 固定        │  宽度: 剩余空间              │
│  背景: #f8f9fa           │  背景: #ffffff               │
│                          │                              │
│  ① 权益初审 ✓            │  权益复审                     │
│  │ 版权审核 2026-03-10    │  ┌──────────────────────┐   │
│  │ 结论: 可用             │  │ 人审结论: ○可用 ○权利瑕疵 │   │
│  │                       │  │ 授权依据: ○RID-001  │   │
│  ② 权益复审 ●（当前）     │  │ ...                  │   │
│  │                       │  │ [提交审查结论]        │   │
│  │                       │  └──────────────────────┘   │
│  ③ 运营授权评估 ○        │                              │
└──────────────────────────────────────────────────────┘
```

- 左侧固定 240px 宽度，右侧自适应
- 中间用竖线（border-left）分隔
- 响应式：≤900px 时改为上下堆叠

#### 3.4.2 左侧时间轴

三个节点纵向排列，每个节点包含：
- **圆点指示器**：
  - 未到达（pending）：灰色空心圆 + 序号数字
  - 已审核（reviewed）：浅蓝色圆 + ✓ 标记
  - 当前活跃（active/pending 且是第一个未审核节点）：深蓝色实心圆 + 外发光 + 序号数字
- **节点连接线**：节点间的竖线（已审核部分为蓝色，未到达为灰色）
- **节点信息**：
  - 节点名称（14px，加粗）
  - 角色描述（12px，灰色）
  - 审核时间（仅已审核显示，12px，灰色）
  - 审核结果标签（仅已审核显示，Tag 形式）

**时间轴节点可点击**：
- 已审核节点和当前活跃节点可点击，切换右侧面板内容
- 未到达节点不可点击（灰色禁用态）
- 点击时高亮选中的节点 + 更新右侧面板标题

#### 3.4.3 三个审核节点定义

| 节点 | ID | 名称 | 角色 | 状态流转 |
|------|----|------|------|---------|
| 1 | node-1 | 权益初审 | 版权审核 | pending → reviewed |
| 2 | node-2 | 权益复审 | 版权审核 | pending → reviewed |
| 3 | node-3 | 运营授权评估 | 运营审核 | pending → reviewed |

**流转规则**：
- 按顺序审核：1→2→3
- 前一节点提交后，下一节点才变为当前活跃节点
- 全部节点审核完成 = 已确权

#### 3.4.4 右侧面板 - 节点一（权益初审）

**可编辑模式**（当前活跃节点为 node-1 时）：

| 字段 | 类型 | 必填 | 选项 | 联动逻辑 |
|------|------|------|------|---------|

1. **授权判断选择「不授权」**：
   - 显示「不授权原因」区域
   - 隐藏「可授权平台」区域 + 清除已勾选平台和日期

2. **授权判断选择「可授权」**：
   - 显示「可授权平台」区域
   - 隐藏「不授权原因」区域 + 清除已选原因

3. **平台 checkbox 联动**：
   - 勾选「快手」→ 下方出现「快手授权起止时间」日期行
   - 勾选「抖音」→ 下方出现「抖音授权起止时间」日期行

   - 两个日期输入框（开始、结束），中间显示 `—` 分隔符
   - 开始日期 min = 今天（过去时间置灰不可选）
   - 开始日期默认值 = max(今天, 权利数据中所有 authStartDate 的最大值)
   - 结束日期 min = 开始日期（动态跟随变化）

#### 3.4.7 提交审查结论

每个节点面板底部有「提交审查结论」按钮。

**提交前校验**（按节点）：

**Node-1**：
1. 人审结论必选
2. 若选「权利瑕疵」→ 瑕疵类型必选

**Node-2**：
1. 人审结论必选
2. 授权依据的引入权利必选
3. 若选「权利瑕疵」→ 瑕疵类型建议选择（继承逻辑）

**Node-3**：
1. 授权判断必选
2. 瑕疵处置结论（条件必填）
3. 若选「不授权」→ 不授权原因必选
4. 若选「可授权」→ 至少选一个平台 + 每个平台的起止日期必填 + 结束日期 > 开始日期

**校验失败**：显示 Toast 提示（warning 类型），定位到未填字段。

**提交成功后**：
1. 更新节点状态为 `reviewed`
2. 记录 reviewer（当前用户）和 reviewTime
3. 重新渲染审核节点区域
4. 自动切换到下一个待审节点（左侧时间轴高亮 + 右侧面板内容）
5. 显示成功 Toast：`{节点名} 审查结论已提交`

#### 3.4.8 修改已审核结论

- 条件：仅当已审核节点的**下一个节点仍为 pending**时，显示「修改」按钮
- 点击修改：将该节点状态临时切回 pending，重新渲染为可编辑面板
- 修改后再次提交即可

### 3.6 合同对照工具

#### 3.6.1 入口

每张权益卡片展开后，左侧字段区域右边缘有一个竖向悬浮的「合同对照」Toggle 按钮：
- 位置：绝对定位在 `rrd-split-left` 右边缘中部
- 默认态：双箭头向左图标 + 竖排「合同对照」文字 + 左侧脉冲动画条
- 激活态：双箭头向右图标 + 竖排「收起」文字 + 蓝色实填背景
- 按钮圆角：左圆角 10px，右圆角 2px（贴边）

#### 3.6.2 分屏布局

点击 Toggle 按钮后：

```
┌─────────────────────┬──────────────────────┐
│  左侧: 权益字段      │  右侧: 合同原文       │
│  50% 宽度            │  50% 宽度             │
│  2列 grid（缩列）    │  合同标题 + 搜索框    │
│  overflow-y: auto    │  合同正文              │
│  滚动查看            │  滚动查看              │
└─────────────────────┴──────────────────────┘
```

**高度对齐机制**：
1. 先激活分屏布局（添加 `rrd-split-active` class）
2. 渲染合同面板内容
3. 在 `requestAnimationFrame` 回调中，测量左侧面板 `scrollHeight`（2列布局后的实际高度）
4. 将此高度设置为 `split-layout` 容器的固定高度
5. 左右面板通过 flex + align-items:stretch 自动等高

#### 3.6.3 合同面板内容

- **Header 区域**：
  - 左侧：合同图标 + 「合同原文」标题
  - 右侧：搜索框（placeholder: `搜索关键词`）

- **正文区域**：
  - 合同原文按 `\n` 分段
  - 合同标题段：居中 + 蓝色底线
  - 条款标题段（以「第X条」开头）：加粗 + 蓝色左竖线
  - 签章行（以「甲方」/「乙方」/「签署日期」开头）：灰色
  - 普通段落：标准正文

#### 3.6.4 关键词搜索高亮

- 输入后 200ms 防抖执行搜索
- 使用 TreeWalker 遍历合同面板内所有文本节点
- 匹配项用黄色高亮（`.contract-search-highlight`）
- 显示匹配数量（`X个匹配` / `无匹配`）
- 自动滚动到第一个匹配项
- 支持 Escape 快捷键清除搜索
- 有清除按钮（×）

#### 3.6.5 关闭分屏

三种方式关闭：
1. 再次点击 Toggle 按钮（toggle 效果）
2. 收起权益卡片时自动关闭
3. 以上均清除分屏高度限制、清空合同面板内容

## 五、数据加载流程

1. 模块加载时，通过 `fetch` 加载 `data.json` 文件
2. 解析 JSON 数据，存储到全局变量 `authData`
3. 调用 `initAuthManagementModule()` 初始化
4. 根据当前 Tab 渲染对应内容
5. 授权审查 Tab 调用 `renderRightsReview()` 渲染列表
6. 点击操作按钮调用 `openRightsReviewDetail(itemId)` 跳转详情
